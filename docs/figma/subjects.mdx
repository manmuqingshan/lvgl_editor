Most of what you design is static: a label reads "72°", a switch is drawn on, a slider
sits at 60%. Those are fine as starting values. But real UIs change while they run: the
temperature updates, the toggle flips, the progress moves.

A **subject** is how you say "this part shows a *live* value." It's a named value in LVGL that your UI
watches; when your firmware changes the subject, every widget bound to it updates on its
own. You can learn more about it in [**Subjects & Data Binding**](../syntax/data-binding#defining-subjects).

<Figure src="/_static/images/figma/subject-binding.png" alt="The Subject Binding fields on a widget's summary card" caption="Give a widget a subject name and it binds to that live value instead of the drawn one." />

## Two kinds of binding

There are two things you can point at a subject, and they solve different problems:

## Bind a value

Select a bindable widget and you'll see a **Subject Binding** row with two
fields:

- **Subject**: the name of the live value (e.g. `room_temp`, `wifi_on`). Typing a name
  is what turns the widget dynamic; leave it blank and the widget keeps its drawn value.
- **Format**: an optional pattern for how the value is shown (e.g. `%d°` renders the
  number `72` as `72°`). See [the format reference](#formatting-the-value) below.

You give **one** subject name; the plugin picks the right LVGL attribute for you based on
what the widget is:

| Widget | What binds to the subject | Value type |
|---|---|---|
| **Label** | its text | string (or a number via **Format**) |
| **Image** | its source (which image shows) | string |
| **Slider**, **Bar** | its value | integer |
| **Switch**, **Checkbox** | its on / off state | boolean |
| **Dropdown**, **Roller** | its selected **index** | integer |

<Callout type="info">
  Subjects come in three kinds (**integer**, **string**, and **boolean**), and the
  plugin infers which from the widget and format. (A boolean is stored as `0` / `1`.)
</Callout>

### Formatting the value

The **Format** field is a small pattern with a placeholder for the value. You'll rarely
need more than the first two:

| Format | Use it for | Value | Shows |
|---|---|---|---|
| `%d` | a whole number | `72` | `72` |
| `%s` | text | `Kitchen` | `Kitchen` |
| `%x` | a number in hexadecimal | `255` | `ff` |
| `%%` | a literal percent sign | (none) | `%` |

Put your own text around the placeholder and it comes through as-is:

| Format | Shows |
|---|---|
| `%d°` | `72°` |
| `%d%%` | `50%` |
| `$%d` | `$5` |
| `%s room` | `Kitchen room` |

<Callout type="info">
  Numbers here are whole numbers (an integer subject) and text is a string. There's no
  decimal format, because subjects hold whole numbers or text, not fractions. Leave
  **Format** blank for plain text with no pattern.
</Callout>

## Bind variants to a subject

This is the powerful one. Say you built a status chip as a Figma component with several
variants (`Charging`, `Standby`, `Fault`), each a different color and icon. Instead of
shipping three components and swapping them by hand, you can collapse the whole set into
**one** component whose look is driven by a single number.

Select the component set and its card offers **Bind States to Subject**. Turn it on, give
the subject a name (or accept the suggested one), and that's it: the card shows a
read-only **Steps** list of how each variant maps to a value.

<Figure src="/_static/images/figma/bind-states.png" alt="Bind States to Subject with its 0-based Steps map" caption="Each variant gets a number, starting at 0: here 0 = Charging, 1 = Standby, 2 = Fault." />

### The mapping is 0-based

Each variant becomes an integer, **starting at 0**, in the order the variants are declared:

| Value | Variant |
|---|---|
| `0` | first variant (`Charging`) |
| `1` | second variant (`Standby`) |
| `2` | third variant (`Fault`) |

<Callout type="warn">
  The index starts at **0**, not 1: `0` is the first variant, `1` the second, and so on.
  At runtime you select one with `lv_subject_set_int(&your_subject, N)`, where `N` is that
  0-based value. The generated file even prints the map (`0=Charging, 1=Standby, …`) in a
  comment so it's easy to reference.
</Callout>

### When you can use it

Variant binding is offered only when the component set has **one** variant axis and its
values are plain names (not state words like Hover/Pressed, which are handled differently;
see below). You need at least two variants. Instances of a bound set are read-only, so they
all follow the shared subject.

## Subjects aren't states

There's a related-but-separate feature worth not confusing with this. Binding *variants to
a subject* (above) is for **content you switch by value**: a status, a mode, a page index.

**States**, such as pressed, checked, and disabled, are different: they're the interaction looks a
widget takes on its own, and you express them by drawing *state variants* (an On switch, a
pressed button). The converter stays faithful there too: it only styles a state if you
drew it. That's covered in [Widget properties](./widget-properties#states).

<Callout type="info">
  Rule of thumb: reach for a **subject** when *your app* decides what shows (a value, a
  mode). Reach for a **state variant** when *the user's interaction* decides it (pressed, on).
</Callout>
